129 /*
130 [2020/7/4] LogicalChildren は、Content にするべきか、子 UITreeItem にするべきか
131 ----------
132 WPF の場合は TreeItem は HeaderdItemsControl であり、Header(Content) と Items は明確に分かれている。
133 
134 Lumino の場合は
135 - カスタマイズをコードファーストで簡単にできるようにしたい
136 - 処理速度のため、レイアウトのためだけのネストを増やしたくない
137 という要求がある。
138 
139 カスタマイズ容易性については、例えば Item の右側に Button をつけたい場合は次のようにしたい。
140 ```
141 treeview1->setGenerateTreeItemHandler([](ln::UITreeItem2* item) {
142  auto button = ln::UIButton::create(u">");
143  button->setSize(20, 20);
144  button->setAlignments(ln::UIHAlignment::Right, ln::UIVAlignment::Center);
145  item->addChild(button); // UIElement デフォルトの addChild フレームワークで追加する場合
146  item->addChild(1, button); // 別のカラムに追加したい場合はこんな感じ
147 });
148 ```
149 
150 処理速度については、もし WPF で同じことをやるなら、Header に StackPanel 追加し、
151 StackPanel の子として TextBlock と Button を追加する必要がある。
152 構造としては正しいだろうけど、これはユーザプログラムとしても面倒 & 処理速度も増える。
153 右揃えするくらいなら UIControl デフォルトのレイアウトでやりたい。
154 
155 こんな感じなので、UITreeView は考え方としては、「ツリーの格好をした ListView」と考えたほうが良い。
156 WPF の TreeItem は子 TreeItem を完全に内包するようにレイアウトも行われるが、それとはまた違った形になる。
157 → そっちの考え方は "UITreeBox" みたいなのをよういしていいかも。
158 古くからある GUIToolKit では "TreeListView" とか呼ばれてそうなもの。
159 
160  */
