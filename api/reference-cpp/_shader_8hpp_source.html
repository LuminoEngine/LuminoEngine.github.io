30 // UniformBuffer, sampler など、Shader の Data を保持する。
31 // 本来は Pass に持たせるべきだが、そうするとユーザープログラムから Material(=Effect) として使いたい場合、
32 // 全く同じ ShaderDefaultDescriptor がたくさん作られることになり、メモリの無駄が多くなる。#136 の問題も出てくる。
33 // そのため ShaderPass と関係を持つのではなく、Shader と関係を持つようにする。
34 /*
35  [2020/12/15] ShaderDefaultDescriptor 計画
36  ----------
37  ShaderDefaultDescriptor = DescriptorSet。
38  正しくやるなら、これは通常、SceneRenderer からのドローコール1つに付き1つのインスタンスが必要になる。
39  それへ、Objectごとに変わるUniformBuffer, Viewごとに変わるUniformBuffer, マテリアルのUniformBufferなどをアタッチして使う。
40  Material 1つ = ShaderDefaultDescriptor 1 つ、ではない。
41  つまり Pool 必須。VulkanDriver とほぼ同じ実装が必要になる。
42 
43  少しずつ移行するなら…
44 
45  まず UniformBuffer を扱えるようにする。
46  現状 m_descriptor->setData() で struct を設定している個所で、SingleFrame な UniformBuffer を
47  GraphicsContext から作り、それ経由でデータをセットするように変更する。
48 
49  極限まで最適化するなら SingleFrame 以外の UniformBuffer もほしいが、ひとまず保留。
50  というか Vulkan の allocationCount の問題があったけど、Buffer 自体はそれほど多く作れるものではない。
51  またMaterial から使うときも、結局 Material 自体は UniformBuffer を持たずに map で値を持っていて、
52  SceneRenderer から submit するときに その map のデータを SingleFrame UniformBuffer へ転記する。
53  ※こうしないと Material のシリアライズが大変。
54 
55  次に ShaderDefaultDescriptor の Pool を作る。
56  必須ではない気もするけど、UniformBuffer は個別にしたのに DescriptorSet は Shader ごとに唯一なインスタンスなのが不自然。
57  RHI 側の肥大化も防止したい。DX12 や Metal 対応の前にやっておきたいところ。
58 
59  ShaderDefaultDescriptor はすべて SingleFrame なオブジェクトなので、GraphicsContext に shader を与えて生成してもらう感じがいいかな。
60 
61  ### Shader 自体が値を保持する必要は？
62 
63  無し、でいいと思う。その役割は Material が負う。
64 
65  でも HC4 では ひとつのShaderをたくさんの Material にセットするとき、シーン内で唯一の意味を持つ値を、
66  全部の Material 側に設定するのがイヤだったから Shder に直接設定していた。
67  実際これはメモリ削減に有効に働いた。手間も少なかったし。
68 
69  「Materialに値が設定されなったときの規定値」として Shader が値を持つのはアリかも。
70 
71  「規定値を設定しておく ShaderDescriptor」を1つ、Shader に持たせる方法がよさそう？
72  ただこの人は UniformBuffer ではなく普通の ByteArray を持つようにしなければならない。
73 
74 
75  SceneRenderer 辺りからの使われ方は、だいたい次のようになる。
76  ```
77  viewUniformBuffer = シーン描画開始時に確定して共有
78 
79  // Subset 単位
80  auto* descriptor = shader->allocateDescriptor();
81  descriptor->setUniformBuffer("LNRenderViewBuffer", viewUniformBuffer);
82  descriptor->setUniformBuffer("LNRenderElementBuffer", elementUniformBuffer);
83  descriptor->setUniformBuffer("LNEffectColorBuffer", effectUniformBuffer);
84  descriptor->setUniformBuffer("$Global", Materialが持つユーザー指定のParams);
85  descriptor->setUniformBuffer("xxxx", Material::setData()で指定されたユーザー指定のstructデータから作ったUB);
86  ```
87  "$Global" と "xxxx" にはデフォルト値を使いたい。
88 
89 
90  [2020/12/16] そもそも ShaderDescriptor をたくさん作れるようにする必要はある？
91  ----------
92  SceneRenderer から本気でマルチスレッドでコマンドリスト作る場合…かな。
93 
94  Vulkan は API の仕様上 Pool からたくさんの Descriptor を作る必要があるけど、
95  ラッパー側では 092f21e7507a6480d61249480ad267bb1916e22f 時点のように
96  単一の固定長 array を持った struct に UniformBuffer や Texture をセットして、
97  submit するだけでかまわない。
98 
99  モダンな GraphicsAPI としては正しいのかもしれないけど、
100  結局 RHI とラッパーで同じようなデータ構造を用意する必要があり、
101  複雑さが上がる一方、メモリ消費・実行速度的に嬉しいことがあまりない。
102 
103  ゲームのように複雑なシーンで動く絵を出すためには、実質ドローコールのたびに
104  Descriptor を作らなければならないためかなりのメモリ消費になる。
105 
106  またラッパー側で GraphicsCommandList をバックバッファの数だけ持つようにして
107  ローテーション組む用になっているが、このときどの GraphicsCommandList と DescriptorPool を
108  対応付けるか、という問題も出てくる。(VulkanDriver と同じような方法で解決せざるを得ないけど)
109  単純に複雑さが上がる。
110 
111 
112  [2020/12/16] それでも ShaderDescriptor が必要？
113  ----------
114  UniformBuffer をサポートしたことで Shader へデータをセットする API が複雑になってきたので、
115  エンジンの 内部から使うものと、外部のユーザープログラムから使うものを区別したい。
116 
117  Unity とかは Shader に対して直接フィールドをセットすることはできないけど、
118  前述の通り直接設定したほうが作業効率が良いこともある。
119 
120  ただ GraphicsContext を使うときにもこれが必要になるので、どのみち公開することになる。
121  Binding への公開はそもそも Graphics 全体の課題なのでいいとしても、
122  Descriptor 自体はある程度本来の使い方に似せたデザインにしたい。
123 
124 
125 */
