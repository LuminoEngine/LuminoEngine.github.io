9 // - WorldObject と AssetModel は 1:1
10 // - View と AssetModel は n:1 (View はインスペクタやプレビューなど様々なところから参照できる)
11 // - 【AssetModel は MVVM でいう ViewModel と Model(save,loadできる)】に相当する。View への変更通知機能を持つ。（強引に分類するなら Model に相当する）
12 // - Prefab の変更を自動的に伝播させたりするのに使う。
13 // - データは AssetModel -> WorldObject の方向にしか書き込まれない。WorldObject は AssetModel に対する readonly な View の位置づけ。
14 // - エディタでは、WorldObject を編集するのではなく、AssetModel を編集することになる。
15 // - Scene と SceneAsset にぶら下がる WorldObject と WorldObjectAsset のツリーは2重管理になる。これは WPF でも View と ViewModel で発生する構図。
16 // - 編集操作はすべて AssetModel 側から行う。Scene のツリーに WorldObject を追加したければ、SceneAsset::createNewObject() とかを呼び出す。
17 // - すべての AssetModel がファイルに保存されるとは限らない。WorldObjectAsset は通常、SceneAsset のファイルに含まれる。プレハブの場合は独立する予定。
18 // - このように WorldObject などの階層構造を表現するため、AssetModel も親子関係を持つ。
19 // - Unity の SerializedObject に相当する。https://anchan828.github.io/editor-manual/web/serializedobject.html
20 // - ※ScriptableObject ではない。
21 // - SerializedObject 同様、編集対象の Object (WorldObject や Component など) をラップし、プロパティを操作する。
22 // - AssetModel を派生して使うことはない。
23 // - 基本的な使い方も SerializedObject と同じく、update -> プロパティset -> apply がワンセット。
24 // - apply で Undo スタックに変更履歴が詰まれる。これにより複数のプロパティをまとめてひとつの undo に入れることも可能。
25 // - Target がプロパティとして持っている Object は、基本的に別 Asset を示す GUID で保存される。
26 // - 別 Asset ファイルに保存したくない場合は、各 Object の serialize を実装する。（RigidBody の Shapes とかはこれになる）
27 // - というか、C++ でリフレクションは相当厳しいので、まずは基本方針 serialize 実装かな。ちなみに Urho3D は Save/Load で行っている。
28 /*
29  [2019/8/18] 保存について
30  ----------
31  AssetModel では保存まで面倒見ないほうがいいかも。
32  このあたりのやり方は Urho3D あたりが参考になるが、ひとまず泥臭くいくしかないかなというところ。
33 
34  なので、ランタイム実行時にシーンロードするときなど、load のためだけに一時的な AssetModel をたくさん作る必要はない。
35  （ルートだけは必要。IDとかをシリアライズする必要があるので）
36
